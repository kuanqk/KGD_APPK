# Бизнес-процесс АППК

## Основной маршрут дела
```
Создание дела
    ↓
Формирование документов
(Извещение о явке + Предварительное решение)
    ↓
Вручение документов НП
    ├── Нарочно (in_person)
    └── Заказным письмом (registered_mail)
         └── Если вернулось (mail_returned):
              ├── Акт налогового обследования
              └── Запрос в ДЭР
    ↓
Назначение заслушивания
    ↓
Проведение заслушивания
    ↓
Оформление протокола
(отсчёт 2 рабочих дней — Celery)
    ↓
Итоговое решение
    ├── Прекращение дела → согласование → terminated
    └── Налоговая проверка → согласование → audit_approved → completed
```

## Полная цепочка статусов (CaseStatus)
```
draft
  ↓ создан документ
notice_created
  ↓ вручение зафиксировано
notice_sent
  ├── delivered (вручено нарочно или заказным)
  └── mail_returned (письмо вернулось)
       ├── act_created (акт оформлен)
       └── der_sent (запрос в ДЭР направлен)
hearing_scheduled (заслушивание назначено)
  ↓
hearing_done (заслушивание проведено)
  ↓
protocol_created (протокол оформлен, отсчёт 2 дней)
  ├── termination_pending → terminated
  └── audit_pending → audit_approved → completed → archived
```

## Жёсткие бизнес-правила
1. Итоговое решение требует протокол заслушивания — без него блокируется
2. После протокола — 2 рабочих дня дедлайн (Celery check_deadlines каждый час)
3. Прекращение только с валидным основанием (TerminationBasis)
4. Signed документы — только новая версия, удаление запрещено
5. Все действия пользователей → AuditLog (обязательно)
6. Данные изолированы по роли через `QuerySet.for_user(user)`
7. Backdating (ввод задним числом) — только с разрешения admin/reviewer
