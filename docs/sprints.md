# АППК — Sprint Plan

## Принципы

- Каждый спринт = 1–2 недели
- Каждый спринт заканчивается рабочей демо-версией
- Приоритет: вертикальные срезы (работающий сквозной путь), а не горизонтальные слои

---

## Sprint 0 — Инфраструктура и скелет (3–5 дней)

**Цель:** Проект запускается локально, CI работает, можно логиниться.

### Задачи

- [ ] Docker Compose: `web`, `db`, `redis`, `worker`, `nginx`
- [ ] Django-проект: создать, подключить PostgreSQL
- [ ] Базовые Django-apps: `accounts`, `cases`, `audit`
- [ ] Модель `User` + роли (Администратор, Оператор, Руководитель, Исполнитель, Наблюдатель)
- [ ] Аутентификация: логин / логаут / смена пароля
- [ ] Middleware для `AuditLog` (вход/выход)
- [ ] Django Admin настроен для базовых моделей
- [ ] `.env.example`, `Makefile` с командами `up`, `migrate`, `test`
- [ ] Базовый шаблон (base.html) с навигацией по ролям

**Критерий готовности:** `docker compose up` → можно войти под разными ролями.

---

## Sprint 1 — Реестр дел и карточка (1–2 недели)

**Цель:** Оператор создаёт дело, видит его в списке, открывает карточку.

### Задачи

- [ ] Модели: `AdministrativeCase`, `Taxpayer`
- [ ] Автогенерация номера дела (формат: `АД-ГГГГ-NNNNN`)
- [ ] Форма создания дела (НП + основание + регион + ответственный)
- [ ] Реестр дел: таблица с фильтрами (статус, регион, ответственный, период)
- [ ] Поиск по: номеру дела, ИИН/БИН, наименованию НП
- [ ] Карточка дела: блок «Общая информация», блок «Данные НП»
- [ ] Лента событий (CaseEvent) — фиксация статусных переходов
- [ ] Разграничение доступа: Наблюдатель видит только свой регион
- [ ] Логирование создания и изменения дела в `AuditLog`

**Критерий готовности:** Оператор создаёт дело → оно появляется в реестре → карточка открывается < 3 сек.

---

## Sprint 2 — Документы и шаблоны (1–2 недели)

**Цель:** Система генерирует процессуальные документы из шаблонов.

### Задачи

- [ ] App `documents`: модели `CaseDocument`, `DocumentTemplate`
- [ ] Версионирование шаблонов (active/archived)
- [ ] Движок подстановки: данные из карточки дела → шаблон → PDF/DOCX
- [ ] Автогенерация номера документа
- [ ] Первые два шаблона: **Извещение о явке**, **Предварительное решение**
- [ ] Блок «Процессуальные документы» в карточке дела
- [ ] Хранение сгенерированного файла (`file_path` + S3/local)
- [ ] Скачивание / предпросмотр документа
- [ ] Запрет удаления документа в финальном статусе

**Критерий готовности:** Оператор нажимает «Сформировать извещение» → получает PDF с реквизитами дела через < 10 сек.

---

## Sprint 3 — Вручение и доставка (1 неделя)

**Цель:** Фиксируем доставку документов НП, обрабатываем возврат.

### Задачи

- [ ] App `delivery`: модель `DeliveryRecord`
- [ ] Два сценария вручения: **нарочно** / **заказное письмо**
- [ ] Поле `tracking_number` + ввод результата доставки
- [ ] Статусный переход: «Направлено НП» → «Ожидается подтверждение»
- [ ] Сценарий возврата: фиксация возврата → смена статуса
- [ ] Блок «Вручение» в карточке дела
- [ ] Реестр отправок/вручений (отдельная страница)

**Критерий готовности:** Дело проходит путь: Извещение → Направлено → Вручено (или Возвращено).

---

## Sprint 4 — Возврат: Акт и Запрос в ДЭР (1 неделя)

**Цель:** При возврате письма доступны обязательные следующие шаги.

### Задачи

- [ ] Шаблон: **Акт налогового обследования**
- [ ] Шаблон: **Запрос в ДЭР об оказании содействия**
- [ ] Бизнес-правило: при статусе «Возврат» → кнопки «Оформить Акт» и «Создать Запрос в ДЭР»
- [ ] Статусные переходы: Возврат → Акт оформлен → Запрос в ДЭР направлен
- [ ] Логирование событий

**Критерий готовности:** После возврата письма оператор формирует акт и запрос без ручного ввода реквизитов.

---

## Sprint 5 — Заслушивание и Протокол (1–2 недели)

**Цель:** Планирование и проведение заслушивания, формирование протокола.

### Задачи

- [ ] App `hearings`: модели `Hearing`, `HearingProtocol`
- [ ] Форма назначения заслушивания: дата, время, место/формат, участники
- [ ] Календарь заслушиваний (простой список с датами)
- [ ] Карточка заседания: участники, ход, результат
- [ ] Шаблон: **Протокол заслушивания**
- [ ] После протокола: старт отсчёта **2 рабочих дня** (Celery-задача)
- [ ] Уведомление ответственным о назначении и приближении срока
- [ ] Бизнес-правило: итоговое решение заблокировано до оформления протокола

**Критерий готовности:** Назначено заслушивание → проведено → оформлен протокол → система отсчитывает 2 рабочих дня.

---

## Sprint 6 — Итоговые решения (1 неделя)

**Цель:** Два сценария завершения дела реализованы полностью.

### Задачи

- [ ] App `decisions`: модель `FinalDecision`
- [ ] **Сценарий A:** Решение о прекращении дела (основания из справочника)
- [ ] **Сценарий B:** Инициирование налоговой проверки + Решение о проведении проверки
- [ ] Шаблоны: Решение о прекращении, Приказ о назначении проверки
- [ ] Обязательные поля: основание, комментарий, ответственный, дата, вложения
- [ ] Статусные переходы к «Прекращено» / «Решение о проверке оформлено»
- [ ] Переход в «Завершено» → «Архив»

**Критерий готовности:** Сквозной процесс от создания дела до любого из двух итоговых решений работает без блокеров.

---

## Sprint 7 — Согласование и возврат на доработку (1–2 недели)

**Цель:** Руководитель согласует, отклоняет или возвращает.

### Задачи

- [ ] App `approvals`: модель `ApprovalFlow`
- [ ] Отправка документа/решения на согласование
- [ ] Экран согласования для Руководителя
- [ ] Три действия: Утвердить / Отказать / Вернуть на доработку
- [ ] Комментарий при возврате (обязательно)
- [ ] Журнал всех резолюций в карточке дела
- [ ] Логирование: кто, когда, с каким комментарием вернул
- [ ] Блокировка перехода в финальный статус без утверждения

**Критерий готовности:** Руководитель видит очередь на согласование, утверждает или возвращает с комментарием.

---

## Sprint 8 — Уведомления и Celery-задачи (1 неделя)

**Цель:** Система сама оповещает о сроках и событиях.

### Задачи

- [ ] App `notifications`: модель `Notification`
- [ ] Celery Beat: периодические задачи проверки сроков
- [ ] Уведомления о: назначении ответственного, приближении срока (за 1 д), просрочке, возврате
- [ ] Внутренние уведомления (bell-иконка в шапке)
- [ ] Email-уведомления (Django Email + SMTP)
- [ ] Страница «Мои уведомления»

**Критерий готовности:** При просрочке срока система автоматически создаёт уведомление ответственному.

---

## Sprint 9 — Отчёты и экспорт (1 неделя)

**Цель:** Аналитика и выгрузки для руководства.

### Задачи

- [ ] Страница «Отчёты и аналитика»
- [ ] 10 типов отчётов из ТЗ (по статусам, регионам, просрочкам и т.д.)
- [ ] Фильтры: период, регион, подразделение, статус, ответственный
- [ ] Экспорт в **PDF** (ReportLab / WeasyPrint)
- [ ] Экспорт в **XLSX** (openpyxl)
- [ ] Оптимизация запросов (select_related, prefetch_related, индексы)

**Критерий готовности:** Отчёт за квартал строится < 15 сек, выгружается в PDF и XLSX.

---

## Sprint 10 — Безопасность, аудит, шлифовка (1 неделя)

**Цель:** Продакшн-готовность системы.

### Задачи

- [ ] HTTPS / TLS-конфигурация Nginx
- [ ] Страница «Логи системы» (только Администратор)
- [ ] Фильтрация логов по пользователю, дате, типу события
- [ ] Страница «Пользователи и роли» — управление из UI
- [ ] Настройки справочников (основания, категории, регионы)
- [ ] Rate limiting, защита от CSRF
- [ ] Резервное копирование БД (cron + pg_dump)
- [ ] Нагрузочное тестирование (25–50 пользователей)
- [ ] Документация: README, API (если нужен), инструкция развёртывания

**Критерий готовности:** Все критерии приёмки из ТЗ (п. 19) пройдены.

---

## Roadmap (визуально)

```
Sprint 0  │ Инфраструктура + Auth
Sprint 1  │ Реестр дел + Карточка
Sprint 2  │ Документы + Шаблоны
Sprint 3  │ Вручение + Доставка
Sprint 4  │ Возврат: Акт + ДЭР
Sprint 5  │ Заслушивание + Протокол
Sprint 6  │ Итоговые решения (A + B)
Sprint 7  │ Согласование + Возврат на доработку
Sprint 8  │ Уведомления + Celery
Sprint 9  │ Отчёты + Экспорт
Sprint 10 │ Безопасность + Шлифовка + Приёмка
```

**Итого:** ~12–16 недель при команде 2–3 разработчика.
